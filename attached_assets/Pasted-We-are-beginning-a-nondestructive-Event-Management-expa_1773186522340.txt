We are beginning a nondestructive Event Management expansion sprint for the Core Platform application. This app is a BetterHelp-style counselor directory and subscription platform with public users/clients, counselors, and administrators. We need to expand the existing event system to support richer event delivery details and future registration/payment/archive capabilities.

IMPORTANT RULES
- Nondestructive only
- Preserve all current event functionality and UI unless expansion requires safe additive changes
- Do not remove any current event features
- Do not redesign unrelated admin areas
- Do not break current public event listing/detail behavior
- Prefer additive schema and UI changes over refactors
- Keep all changes consistent with the existing design system, routing patterns, storage layer, schema conventions, and admin patterns

PRIMARY GOAL
Expand the event data model and admin event management capabilities to support:
- virtual meeting join links
- dial-in information
- recording/video links for past events
- free vs paid events
- registration enablement
- registration windows
- capacity and waitlist readiness
- event lifecycle/status controls
- visibility/access controls
- speaker/host metadata
- map-ready physical location behavior
This phase should prepare the foundation for later registration, Stripe payment, and recording archive phases, but should NOT yet implement the full registration or payment workflows.

REQUIRED FEATURE FIELDS / CAPABILITIES
Add or safely extend event support for fields such as:
- isVirtual
- virtualJoinUrl
- virtualDialInInfo
- recordingUrl
- hasRecording or equivalent computed/support field
- registrationEnabled
- registrationType or equivalent (free / paid)
- registrationFee
- registrationCurrency
- registrationOpensAt
- registrationClosesAt
- capacity
- waitlistEnabled
- status (draft / published / canceled / completed)
- visibility (public / members-only / counselors-only / admins-only or equivalent role-based model that fits current architecture)
- timezone
- locationName
- locationAddress
- latitude
- longitude
- speakerName / hostName
- speakerBio / hostBio
- speakerImage or host image support if practical and already aligned with current media handling patterns
Do not overbuild beyond what is clearly needed for the upcoming phases.

IMPLEMENTATION REQUIREMENTS
1. Inspect:
   - current event schema
   - current event admin CRUD
   - public event listing/detail rendering
   - shared event types
   - any storage/service logic related to events
   - existing admin forms/components for events
   - current map implementation patterns from counselor details pages

2. Extend the event schema safely and create any required migrations in a nondestructive way.

3. Update the admin event create/edit flows so admins can manage all newly added event fields in a common-sense way.

4. Ensure admin UX remains clean and organized. Group fields logically, such as:
   - basic event info
   - scheduling/timezone
   - location / virtual attendance
   - registration settings
   - pricing
   - visibility/status
   - recording/archive info
   - host/speaker details

5. Preserve existing event cards/listing behavior. Do not yet expose unfinished registration/payment/archive features publicly unless the data is safe to show.

6. Ensure event detail data contracts are forward-compatible with upcoming phases.

7. If there are existing shared types or API contracts for events, update them carefully to avoid drift between client/server/shared schema types.

8. Add validation for obvious event field constraints such as:
   - paid events should require a valid registration fee
   - virtual join links should validate as URLs where practical
   - registration close should not precede registration open
   - event end should not precede event start
   - completed/archive-related behavior should remain sensible

9. Keep the implementation additive. Do not rewrite the whole event module.

NONDESTRUCTIVE RULES
- No route removals
- No deletion of old event data
- No redesign of unrelated public/admin pages
- No registration implementation yet beyond admin model support
- No Stripe checkout implementation yet
- No archive browsing implementation yet

VALIDATION CHECKLIST
Before finishing:
1. Typecheck/check passes
2. Build passes
3. Existing event pages still work
4. Admin can create and edit events with the new fields
5. Existing events without the new fields still render safely
6. Any schema migrations are safe and non-destructive
7. API contracts remain consistent

REQUIRED OUTPUT
At the end provide:
1. Summary of schema/admin event enhancements
2. Files changed grouped by area
3. Migration notes
4. Any follow-up recommendations for the next event phases
5. Docs update

DOCS UPDATE REQUIREMENT
Update project documentation to include:
- new event model fields
- admin event configuration behavior
- validation rules
- future phase dependencies for registration/payment/archive features

Work carefully and complete the full pass in one implementation cycle.
