We are beginning Phase 4 of the Core Platform CMS sprint: seeding the existing public-facing website pages into CMS and safely integrating CMS-based frontend rendering.

CONTEXT
The goal is that existing public website pages should be available inside CMS and ready for admin editing. This should be done with a practical seeding/import strategy, not unrealistic automatic reverse engineering. Existing public pages should continue to work safely, with fallback behavior where needed.

IMPORTANT RULES
- Nondestructive only
- Preserve current public routes and SEO-critical slugs
- Use a practical import/seeding approach
- Keep safe fallback behavior for pages not fully CMS-driven yet
- Only target public-facing website pages, not authenticated app screens
- Preserve current visual quality and UX

PRIMARY GOAL
Seed existing public-facing website pages into CMS records and integrate public rendering so CMS-managed page content can begin powering the live website safely.

IMPLEMENTATION REQUIREMENTS
1. Inspect:
   - current public routes
   - current public page components
   - CMS pages and builder content structures
   - current blog/public content relationships
   - any shared SEO or metadata handling

2. Identify the public-facing pages that should belong in CMS v1, such as:
   - Home
   - About
   - Contact
   - informational/marketing pages
   - content landing pages
   - any resource-like public pages
   Do not include authenticated application screens.

3. Create a seeding/import strategy that:
   - creates CMS page records for current public pages
   - populates title/slug/template/source metadata
   - seeds builder content with practical starter blocks that reflect the existing page structure as closely as reasonably possible
   - does not attempt impossible perfect reverse engineering

4. Integrate public rendering so that:
   - when a CMS-managed version exists and is published, the frontend can render it safely
   - where no safe CMS-managed version exists yet, current route rendering can continue as a fallback
   - route behavior remains stable

5. Ensure CMS-rendered blocks use the new media handling approach for images/backgrounds, not raw image URL assumptions.

6. Preserve existing layout, design language, and SEO structure as much as possible.

7. Make admin editing practical immediately after seeding.

NONDESTRUCTIVE RULES
- No deletion of current public pages
- No breaking route slugs
- No broad router rewrite
- No migration of dashboards/authenticated app flows into CMS
- No degradation of current SEO behavior

VALIDATION CHECKLIST
1. Typecheck/check passes
2. Build passes
3. Seeded CMS page records exist for target public pages
4. Published CMS pages render safely
5. Fallback route behavior works where needed
6. Existing public routes remain stable
7. Admin can open seeded pages in CMS and edit them

REQUIRED OUTPUT
Provide:
1. Summary of seeding/import and rendering integration
2. Files changed
3. Page-by-page seeding notes
4. Any fallback behavior details
5. Follow-up recommendations
6. Docs update

DOCS UPDATE REQUIREMENT
Update docs to include:
- which pages are CMS-managed
- seeding/import strategy
- rendering/fallback behavior
- public route integration notes
