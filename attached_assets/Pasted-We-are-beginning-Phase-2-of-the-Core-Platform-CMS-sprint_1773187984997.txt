We are beginning Phase 2 of the Core Platform CMS sprint: building the visual page builder foundation and block registry for CMS-managed public pages.

CONTEXT
CMS foundation now exists with admin navigation and page records. We need a modern page builder experience for public-facing website pages, inspired by Webflow-style editing but implemented safely within the Core Platform architecture.

IMPORTANT RULES
- Nondestructive only
- Preserve current public page rendering until CMS rendering is safely wired
- Reuse/adapt existing Core Platform sections/components wherever practical
- Do not attempt pixel-perfect Webflow cloning
- Keep the builder structured, typed, and maintainable
- Anywhere a builder block would normally ask for image URLs or background image URLs, do NOT use raw URL inputs. Those must become media picker/dropzone upload patterns backed by Cloudflare R2 in the next phase-compatible way.

PRIMARY GOAL
Implement a modern structured CMS page builder with:
- builder editor UI
- block registry
- add/edit/remove/reorder blocks
- block property editing
- preview-safe rendering in admin
- future compatibility with public page rendering
- future compatibility with reusable sections and media library

IMPLEMENTATION REQUIREMENTS
1. Inspect:
   - CMS page model
   - current public page sections/components
   - existing Core Platform design patterns
   - any home/about/events/blog/resource content sections
   - current admin UI conventions

2. Add a builder editor experience to CMS page editing.

3. Create a typed block registry based on real Core Platform page patterns. Start with a sensible block set such as:
   - Hero
   - Rich Text
   - Text + Image
   - CTA
   - FAQ
   - Testimonials
   - Cards Grid
   - Featured Counselors
   - Events Preview
   - Blog Preview
   - Button Group
   - Divider / Spacer
   - Contact / Info Section
   - Video Embed
   - Section Header
   - Image / Media Block
Do not overbuild unnecessary blocks in this phase.

4. Each block should have:
   - typed props/schema
   - default values
   - admin controls
   - renderer compatibility
   - safe fallback behavior

5. Store page-builder content in structured JSON or equivalent.

6. Support:
   - add block
   - remove block
   - reorder block
   - edit props
   - save draft
   - preview inside admin

7. For image-related block props, do not leave raw image URL entry as the intended final UX.
If temporary internal fields exist, they must clearly be designed for replacement by the dedicated R2 upload/media picker flow in the next phase.

8. Keep the renderer reusable for future public-page output.

NONDESTRUCTIVE RULES
- No live public route replacement yet
- No removal of current hardcoded public pages yet
- No blog migration yet
- No media-library overbuild in this phase

VALIDATION CHECKLIST
1. Typecheck/check passes
2. Build passes
3. Builder loads correctly
4. Admin can add/edit/reorder/remove blocks
5. Builder content saves correctly
6. Admin preview renders safely
7. Existing public pages remain stable
8. Shared types remain consistent

REQUIRED OUTPUT

1. Summary of builder/editor work

3. Block registry notes
4. Known limitations and follow-up recommendations
5. Docs update

DOCS UPDATE REQUIREMENT
Update docs to include:
