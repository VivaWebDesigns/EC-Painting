We are beginning Phase 3 of the CMS > SEO sprint for Core Platform: implementing structured data support for optimal search engine understanding and eligibility for rich results.

CONTEXT
Google recommends using JSON-LD for structured data and requires markup to match the visible main content of the page. Structured data does not guarantee rich results, but it helps Google understand page content and can improve eligibility for enhanced search appearances. Core Platform has multiple public content types that should support structured data, including:
- sitewide organization identity
- public page hierarchy/breadcrumbs
- blog/article content
- event pages
- event recordings/video pages where appropriate
- FAQ sections where real FAQ content is visibly present

IMPORTANT RULES
- Nondestructive only
- Preserve current page/blog/event behavior
- Use JSON-LD as the primary structured data format
- Only output structured data that truthfully matches visible page content
- Do not add misleading, hidden, or inapplicable schema markup
- Keep generation modular, typed, and page-aware
- Any image references used in structured data should come from the actual CMS/R2-backed media assets where applicable

PRIMARY GOAL
Build a structured data generation system for the public site and CMS-managed content.

REQUIRED STRUCTURED DATA TARGETS
Implement support for the most relevant structured data types for Core Platform, including:
1. Organization
2. BreadcrumbList
3. Article for blog/editorial pages
4. Event for event detail pages
5. VideoObject for event recordings or video-content pages where appropriate
6. FAQPage only when there is a visible FAQ section on the page
Do not overbuild unrelated schema types.

IMPLEMENTATION REQUIREMENTS
1. Inspect:
   - global SEO settings
   - CMS pages
   - blog content model
   - event pages and event model
   - any FAQ-style content blocks
   - current public rendering/head injection patterns
   - breadcrumb/page hierarchy behavior

2. Create a modular structured data engine that can generate JSON-LD per page type.

3. Add Organization structured data using sitewide/global SEO/org settings where appropriate.

4. Add BreadcrumbList structured data for relevant public pages and posts where hierarchy is known.

5. Add Article structured data to blog/article pages, using the content’s actual visible fields and metadata.

6. Add Event structured data to event detail pages using real event details already shown on the page.

7. Add VideoObject structured data where a page prominently includes a recording or video and the visible page content supports it.

8. Add FAQPage structured data only when the page visibly includes actual FAQ content and the markup accurately reflects that content.

9. Ensure structured data image references use real CMS-managed/R2-backed media assets where appropriate.

10. Make the structured data architecture extensible for future content types, but do not overbuild unsupported or unnecessary schema types.

11. Keep output safe, valid, and tied to public-facing content only.

NONDESTRUCTIVE RULES
- No fake or speculative schema fields
- No hidden-content schema
- No schema types that do not match the actual page content
- No overengineering beyond current site needs
- No breaking existing head/meta behavior

VALIDATION CHECKLIST
1. Typecheck/check passes
2. Build passes
3. Organization JSON-LD renders correctly where appropriate
4. Breadcrumb JSON-LD renders correctly where appropriate
5. Article JSON-LD renders correctly for blog posts
6. Event JSON-LD renders correctly for event pages
7. VideoObject JSON-LD renders correctly only where appropriate
8. FAQPage JSON-LD appears only when the visible page content qualifies
9. Structured data uses real content fields and media assets
10. Public pages remain stable
