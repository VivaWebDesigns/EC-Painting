We are beginning a nondestructive platform integrity pass for the Core Platform application. This app is a production-oriented BetterHelp-style counselor directory and subscription platform built with React + TypeScript on the frontend and Express + TypeScript + Drizzle on the backend. Your task is to stabilize the codebase so future phases can be implemented safely.

IMPORTANT CONTEXT
- This project already has working features and existing UI/UX that must be preserved.
- Do not redesign the app.
- Do not remove features.
- Do not rename routes, API contracts, environment variables, schema entities, or components unless absolutely required to resolve a concrete type or build issue.
- Do not perform destructive refactors.
- Prefer the smallest safe change that restores consistency.
- Preserve all current behavior unless a bug fix is required to make existing intended behavior work correctly.

PRIMARY GOAL
Complete a full platform integrity and TypeScript recovery pass so the project can pass validation cleanly and serve as a safe foundation for future improvements.

PRIORITIES
1. Resolve all current TypeScript errors across client, server, shared schema/types, and utilities.
2. Fix shared contract mismatches between API responses, storage methods, React components, route params, and schema definitions.
3. Install and configure any missing typings/packages required for type safety.
4. Ensure validation scripts are present and usable.
5. Preserve all current functionality and UI behavior.
6. Document all changes clearly.

KNOWN CONTEXT FROM PRIOR REVIEW
- TypeScript currently fails.
- There are route param typing issues.
- There is at least one missing type package such as @types/sanitize-html.
- Shared contracts between UI/data models/realtime/storage are inconsistent.
- Build now correctly hard-gates on typecheck, which is good and must remain.
- This is a stabilization pass, not a feature expansion pass.

IMPLEMENTATION REQUIREMENTS
1. Start by inspecting:
   - package.json
   - tsconfig files
   - server entry and route registration
   - shared schema/types
   - storage layer
   - major client feature modules
   - any existing validation/build scripts

2. Run the project’s existing typecheck/check command and use the errors as the source of truth.

3. Fix TypeScript errors safely across:
   - route handler signatures
   - Wouter route params if typed incorrectly
   - React component prop contracts
   - shared API response types
   - storage return types
   - Drizzle schema/type inference usage
   - utility package typings
   - event handler signatures
   - nullable vs optional field mismatches
   - enum/union mismatches
   - any bad assumptions between client and server models

4. If a shared type is duplicated in multiple places and causing drift, consolidate carefully so both client and server consume the same source of truth where practical.

5. If validation scripts are missing or incomplete, safely add:
   - a working lint script if lint tooling already exists or can be added cleanly
   - a test script placeholder only if it can be added without breaking the current repo
   - do not create noisy or speculative tooling changes beyond what is necessary for a stable foundation

6. Ensure the build/typecheck gate remains enforced. Do not weaken validation in order to “make it pass.”

7. Do not change runtime behavior unless required to fix a clearly broken implementation revealed by type issues.

8. Do not introduce new major dependencies unless required for correctness. Small type/support packages are acceptable.

9. Keep all changes domain-consistent with the current app structure.

NONDESTRUCTIVE RULES
- No database resets
- No schema deletions
- No removal of existing routes or pages
- No UI redesign
- No feature removals
- No replacing whole subsystems
- No weakening auth, validation, or build checks
- No broad “any” typing as a shortcut unless there is no safer option, and if used, keep it minimal and localized with a comment explaining why

QUALITY BAR
- Prefer precise types over broad casts
- Prefer fixing root-cause contract mismatches over suppressing errors
- Avoid introducing tech debt while fixing tech debt
- Maintain readability and consistency with the current codebase

VALIDATION CHECKLIST
Before finishing:
1. Run the typecheck/check command and confirm it passes.
2. Run build if available and confirm it succeeds.
3. Run lint if configured.
4. Verify no major user-facing flows were broken by the fixes:
   - public directory load
   - therapist dashboard access
   - admin dashboard access
   - login/logout flow
   - events load
   - messaging-related types compile cleanly
5. Summarize all fixes by category.

REQUIRED OUTPUT
At the end, provide:
1. A concise summary of what was fixed
2. A list of files changed grouped by area
3. Any remaining non-blocking risks or follow-up recommendations
4. Suggested next phase after this one
5. A docs update

DOCS UPDATE REQUIREMENT
Create or update project documentation to include:
- what integrity/type fixes were made
- why they were needed

Work carefully and complete the full pass in one implementation cycle.
