We are beginning Phase 6 of the Core Platform Event Management sprint: adding event admin workflow polish and practical event management tools.

CONTEXT
The core event model, public details, registration, paid registration, and recording archives are now in place or underway. We need to round out the event management system with the practical tools admins need to operate events effectively.

IMPORTANT RULES
- Nondestructive only
- Preserve all existing event/admin behavior
- Do not redesign unrelated admin systems
- Prefer practical, maintainable workflows over overly complex automation

PRIMARY GOAL
Add sensible event management tools such as:
- capacity tracking improvements
- waitlist support if not already fully implemented
- attendance tracking / check-in support
- event status lifecycle improvements
- duplicate event / copy event support
- reminder/follow-up email workflow support
- basic event analytics summary
- cancellation handling groundwork

IMPLEMENTATION REQUIREMENTS
1. Inspect current event admin flows, registration records, and email template system.

2. Implement or finalize practical tools including:
   - capacity usage/remaining seats visibility
   - waitlist behavior if capacity is exceeded and waitlistEnabled is true
   - attendee check-in / attended tracking
   - no-show vs attended support if practical
   - duplicate/copy event action to speed up event creation
   - admin-friendly status controls and lifecycle labels
   - reminder email hooks/templates for upcoming events
   - post-event recording available/follow-up templates where aligned with the current system
   - basic event analytics summaries such as registrations, attendance count, revenue for paid events if safely available, archive availability, etc.

3. Keep analytics simple and admin-usable. Do not build an enterprise BI dashboard in this phase.

4. If cancellation handling is introduced, keep it safe and minimal:
   - event canceled state
   - admin notification/user communication support
   - registration status updates where needed
Do not build a complex self-service refund portal unless it is already trivial and safe within the current architecture.

5. Keep all admin workflows consistent with the existing admin UI conventions.

NONDESTRUCTIVE RULES
- No broad admin redesign
- No overbuilt automation engine
- No destructive event/registration changes
- No breaking the existing Stripe/event flows

VALIDATION CHECKLIST
1. Typecheck/check passes
2. Build passes
3. Admin can manage event capacity/attendance sensibly
4. Duplicate event works safely
5. Reminder/follow-up workflow support is functional
6. Basic event analytics display correctly
7. Existing event flows remain stable

REQUIRED OUTPUT
Provide:
1. Summary of admin workflow enhancements
2. Files changed
3. Analytics/attendance/capacity notes
4. Any remaining recommended future event improvements
5. Docs update

DOCS UPDATE REQUIREMENT
Update docs to include:
- event admin workflow enhancements
- attendance and capacity behavior
- duplicate event behavior
- reminder/follow-up workflow notes
- analytics summary behavior
