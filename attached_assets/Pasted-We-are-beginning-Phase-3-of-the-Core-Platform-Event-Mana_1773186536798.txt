We are beginning Phase 3 of the Core Platform Event Management sprint: implementing a nondestructive free event registration system for eligible users.

CONTEXT
Core Platform already has users, counselors, admins, events, email templates, and admin systems. We now need to allow event registration for free events, give admins visibility into registrants, support CSV export, and integrate registration-related email workflows. This phase should focus on free registration only. Paid registration via Stripe will be handled in the next phase.

IMPORTANT RULES
- Nondestructive only
- Preserve existing event behavior
- Do not redesign unrelated auth or admin systems
- Do not implement paid Stripe event checkout in this phase
- Prefer authenticated registration unless the current architecture clearly supports a safe guest model; if guest support is not already clearly established, default to authenticated user registration only
- Keep data contracts forward-compatible for the paid registration phase

PRIMARY GOAL
Implement free event registration workflows including:
- event registration enablement logic
- user registration for eligible free events
- admin registrant management
- CSV export of registrants
- common-sense storage of event registrations in admin
- registration-related email templates and trigger wiring

EXPECTED FUNCTIONALITY
1. Users who are allowed to register can register for eligible events.
2. Admins can view registrants for each event in a clear and usable way.
3. Admins can export registrants to CSV.
4. Registration-related email templates/workflows exist for common cases.
5. Capacity logic can be respected if capacity is configured.
6. Registration-open and registration-closed windows are enforced.
7. The data model should support future payment tracking without requiring rework.

IMPLEMENTATION REQUIREMENTS
1. Inspect:
   - current event model
   - current users/auth model
   - admin event management UI
   - email template system
   - CSV export patterns if already present elsewhere
   - any existing event detail actions/buttons
   - role/permission model for users/counselors/admins

2. Add the necessary event registration schema/model/storage/API support, such as:
   - event registrations table/entity
   - eventId
   - userId (or guest identity only if clearly supported safely)
   - fullName
   - email
   - optional phone if aligned with current patterns
   - registration status
   - attendance-related future-ready fields if useful
   - timestamps
   Keep it clean and not overbuilt.

3. Implement registration rules for free events:
   - event must be published/visible appropriately
   - registrationEnabled must be true
   - registration type must be free for this phase
   - registration window must be open if configured
   - capacity must be respected if configured
   - duplicate registration should be prevented sensibly

4. Add public/authenticated event detail actions for registration where appropriate.

5. Add admin registrant views in a common-sense location, likely within event management and/or event detail admin pages.

6. Add CSV export for event registrants with sensible columns.

7. Add or extend email templates/workflows for common event registration scenarios such as:
   - registration confirmation
   - registration canceled if supported
   - registration waitlisted if waitlist is already implemented safely
   - admin notification if appropriate and already aligned with current notification/email patterns
Do not over-automate beyond what fits the current system cleanly.

8. Keep the UI consistent with the current design system.

9. Prepare contracts so the next phase can add payment status and Stripe references without breaking this structure.

NONDESTRUCTIVE RULES
- No paid Stripe registration in this phase
- No broad event redesign
- No unrelated auth rewrites
- No removal of existing event/admin features

VALIDATION CHECKLIST
1. Typecheck/check passes
2. Build passes
3. Eligible users can register for free events
4. Duplicate registrations are handled properly
5. Capacity/window rules work correctly
6. Admin can view registrants
7. CSV export works
8. Email templates/workflows are wired correctly
9. Existing non-registration events still behave safely

REQUIRED OUTPUT
Provide:
1. Summary of registration features added
2. Files changed grouped by area
3. Registration data contract details
4. CSV export behavior summary
5. Email workflow summary
6. Follow-up recommendations for paid registration
7. Docs update

Update docs to include:
