We are beginning Phase 2 for the Core Platform application: a nondestructive security hardening pass. This app is a counselor directory and subscription platform with authentication, counselor approval, Stripe subscriptions, messaging, email, and admin operations. Your task is to harden the application for production without changing the existing UX or removing features.

IMPORTANT RULES
- Nondestructive only
- Preserve all current flows and UI behavior
- Do not redesign authentication
- Do not remove any routes or features
- Do not introduce breaking API changes unless strictly required for security, and if so, keep them minimal and backward compatible where possible
- Do not disable convenience behavior in development unless specifically scoped to non-production enforcement

PRIMARY GOAL
Improve security posture around secrets, auth, cookies, reset flows, logging, and request protection.

PRIORITIES
1. Enforce required secrets in non-development environments
2. Add rate limiting / brute-force protection to login and password reset related endpoints
3. Add CSRF or strict origin-based protections for state-changing cookie-auth routes
4. Add security headers middleware
5. Add request payload size protections where appropriate
6. Redact sensitive information from logs
7. Preserve current UX and admin workflows

IMPLEMENTATION REQUIREMENTS
1. Inspect current auth middleware, auth routes, password reset flow, server bootstrap, logger behavior, and any config/service initialization.
2. Remove insecure fallback behavior in non-development environments for secrets such as session/auth/encryption keys.
3. Add safe production enforcement:
   - fail fast outside development if critical secrets are missing
   - keep development ergonomics reasonable
4. Add rate limiting and anti-abuse protections for:
   - login
   - forgot password
   - reset password
   - any other obvious auth-sensitive endpoints
5. Add protection for cookie-auth mutation routes:
   - CSRF token strategy or
   - strict origin/referer validation for same-site trusted origins
   Choose the least disruptive approach that fits the current architecture.
6. Add security headers middleware such as helmet or equivalent safe configuration.
7. Add body size limits and any other obvious request hardening where needed.
8. Review logging and ensure sensitive fields are not logged:
   - passwords
   - reset tokens
   - auth/session secrets
   - email content where sensitive
   - potentially sensitive counselor/client message payloads
9. Keep Stripe, Mailgun, R2, and admin flows working as they do now.

NONDESTRUCTIVE RULES
- No auth flow redesign
- No UI redesign
- No route removals
- No weakening current functionality
- No breaking local development unnecessarily

VALIDATION CHECKLIST
1. Typecheck passes
2. Build passes
3. Login still works
4. Password reset flow still works
5. Counselor/admin session flows still work
6. Protected routes still behave correctly
7. Rate limits apply appropriately
8. Missing secret enforcement only affects non-dev as intended

REQUIRED OUTPUT
Provide:
1. Summary of security improvements
2. Files changed
3. Any environment variables added/required
4. Any follow-up recommendations
5. Docs update

DOCS UPDATE REQUIREMENT
Update docs to include:
- new security protections
- environment requirements
- rate limit behavior
- request protection strategy
- logging/redaction decisions
