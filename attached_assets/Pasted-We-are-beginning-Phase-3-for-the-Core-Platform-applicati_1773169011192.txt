We are beginning Phase 3 for the Core Platform application: moving therapist directory search/filter logic from client-heavy in-memory processing to server-side query semantics with pagination, while preserving the existing public directory UX.

CONTEXT
Core Platform is a BetterHelp-style counselor directory for Core Platform-informed counselors. The therapist directory is a core product surface and must scale beyond the current seed data. Prior review found that the backend currently accepts some search/filter params, but the storage layer only partially applies them server-side, leaving the client to do too much filtering/sorting in memory.

PRIMARY GOAL
Implement robust server-side directory filtering, searching, sorting support where appropriate, and pagination metadata, while preserving the current public UI and URL-driven deep-link behavior.

IMPORTANT RULES
- Nondestructive only
- Preserve the existing public directory experience
- Preserve current filter UI and URL param behavior
- Do not redesign the map/list layout
- Do not remove existing client-side convenience filtering unless fully replaced safely
- Keep backward compatibility where practical

REQUIREMENTS
1. Inspect:
   - therapist directory route(s)
   - therapist storage/query layer
   - public directory page/component logic
   - current URL param/deep-link filter handling
   - map/list data usage

2. Move primary filtering/search semantics into the backend for at least:
   - search text
   - specialization
   - language
   - practice mode
   - accepting clients
   - approval/active visibility requirements
   Include any other current filters that already exist in UI/state.

3. Implement pagination response shape. Use either:
   - page/pageSize/total/items
   or
   - cursor/nextCursor/items/total when appropriate
   Choose the least disruptive option for the current codebase.

4. Ensure the client consumes paginated/filter-driven API results correctly.

5. Preserve current deep-link URL filter support.

6. Keep client-side filtering only for local UI convenience if needed, but not as the primary source of truth.

7. Make sure featured/home page and any other therapist listing views are not broken.

8. Avoid overengineering. This phase is not full-text search infrastructure unless already easy and safe within the current DB model.

NONDESTRUCTIVE RULES
- No redesign of public pages
- No removal of current filters
- No breaking current therapist card/profile behavior
- No destructive schema changes unless absolutely required and safe

VALIDATION CHECKLIST
1. Typecheck passes
2. Build passes
3. Directory loads correctly
4. Filters work correctly via URL params and API
5. Pagination works correctly
6. Empty states and no-results states still behave properly
7. Existing therapist profile links still work
